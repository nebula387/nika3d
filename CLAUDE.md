# Nika 3D — голосовой 3D-компаньон

## Кто использует этот проект

Иван, личный pet-проект. Уровень английского ограничен — комментарии в коде,
сообщения в консоли и объяснения должны быть на русском, если явно не сказано
иначе. Опыта с Unity/Godot/C#/GDScript нет, начинаем с нуля. С Python знаком.

Claude Code должен объяснять шаги установки и настройки подробно,
не предполагать знание игровых движков. Когда что-то нужно сделать руками
вне кода (установить программу, нажать кнопку в GUI) — давай пошаговую
инструкцию, не просто "настройте Godot".

## Что строим

Голосовой 3D-компаньон по имени Ника:
- Активация по имени ("Ника" / "Nika"), как в текущем прототипе
- Распознавание речи, LLM-диалог, синтез голоса — как раньше
- НО: вместо Live2D-аватара (VTube Studio) — полноценный 3D-персонаж
- Персонаж не копирует пользователя (это не VTube/трекинг лица),
  а действует самостоятельно как игровой NPC
- Голосовой разговор = просто лицо/липсинк говорит
- По запросу пользователя — персонаж может двигаться/жестикулировать всем телом
- Долгосрочная цель: NPC-компаньон с расширяемым набором действий
  (жесты, перемещение, взаимодействие со сценой)

## Железо пользователя

- Windows 11
- GPU: Intel Arc A580, 8GB VRAM (НЕ Nvidia — это критично!)
- RAM: 16GB

## КРИТИЧЕСКОЕ ограничение архитектуры

GPU — Intel, не Nvidia. Это означает:
- НЕЛЬЗЯ использовать что-либо, требующее CUDA (Nvidia Audio2Face,
  TensorRT, любые CUDA-only ML библиотеки)
- Локальная LLM на этой карте — не вариант для быстрой и качественной
  работы, поэтому LLM идёт через облачный API
- Для лицевой анимации/липсинка — только CUDA-независимые решения
  (Rhubarb Lip Sync — CPU, или ONNX Runtime с DirectML backend)
- Рендеринг 3D-сцены в Godot — должен быть на профиле Compatibility/Mobile
  или умеренном Forward+, без тяжёлых эффектов, рассчитанных на топовые GPU

Если в любой момент предлагается решение — первым делом проверяй,
не тянет ли оно скрытую CUDA-зависимость.

## Архитектура: два процесса, один мост

Система состоит из ДВУХ независимых программ, которые общаются по сети:

### 1. Python — "мозг" (папка `brain/`)
Вся логика, которую Иван будет читать и свободно менять:
- Запись микрофона + распознавание речи (Whisper)
- Определение слова активации "Ника"
- Запрос к облачному LLM (провайдер — см. ниже, выбирается в начале работы)
- Парсинг ответа LLM: текст для озвучки + (опционально) команда действия
- Синтез речи (Piper TTS, как в старом проекте)
- Генерация визём (форм рта) из WAV через Rhubarb Lip Sync
- Отправка команд в Godot через WebSocket-сервер (Python — сервер,
  Godot — клиент, либо наоборот — решить на этапе 1, см. ниже)

### 2. Godot — "тело" (папка `godot_project/`)
Тонкий исполнитель, минимум логики, почти не меняется после настройки:
- 3D-сцена с rigged-персонажем (источник модели — Mixamo, см. инструкции)
- Получает JSON-команды по WebSocket
- Переключает Animator/AnimationPlayer между состояниями (Idle/Talking)
- Применяет визёмы к blend shapes/морф-таргетам рта
- Проигрывает анимации действий по имени (wave, nod, etc.)

### Протокол моста (JSON по WebSocket)

Формат сообщений Python → Godot нужно зафиксировать в `docs/protocol.md`
и обновлять при любом изменении. Стартовый набросок:

```json
{
  "type": "speech_state",
  "state": "talking"
}
```

```json
{
  "type": "visemes",
  "data": [{"time": 0.0, "shape": "X"}, {"time": 0.12, "shape": "B"}]
}
```

```json
{
  "type": "action",
  "name": "wave"
}
```

Точную схему дорабатывать в коде, но docs/protocol.md должен всегда
отражать актуальный формат — это контракт между двумя независимыми
кодовыми базами.

## Поэтапный план (НЕ перепрыгивать этапы)

### Этап 0 — выбор LLM-провайдера и окружение ✅ ГОТОВО
Groq API, модель `openai/gpt-oss-120b`. Ключ в `.env`.

### Этап 1 — минимальный вертикальный срез ✅ ГОТОВО
Персонаж стоит в Godot-сцене (Quaternius Casual.fbx, анимации CharacterArmature|Idle
и CharacterArmature|Wave). Иван говорит "Ника", Python распознаёт, шлёт текст в LLM,
получает ответ на английском, озвучивает через Piper TTS. Во время озвучки Python
шлёт `speech_state: talking` → Godot переключает анимацию на Wave; после —
`speech_state: idle` → возврат в Idle. Режим разговора: 10 секунд ожидания
продолжения без слова активации. Весь пайплайн работает и проверен.

### Этап 2 — липсинк по визёмам
Добавить Rhubarb, генерировать визёмы из WAV, прокидывать в Godot,
управлять blend shapes рта покадрово.

### Этап 3 — команды действий
LLM начинает возвращать структурированный ответ (текст + опциональное
поле action), Python парсит и шлёт action в Godot, тот проигрывает
нужную анимацию из библиотеки Mixamo.

### Этап 4+ — навигация, взаимодействие со сценой (не проектировать
заранее в деталях, обсуждать когда дойдём)

Не пытайся реализовать несколько этапов сразу одним большим коммитом.
После каждого этапа — рабочая, тестируемая версия, и пауза для проверки
Иваном перед тем как двигаться дальше.

- Объясняй технические термины игровых движков (Animator, blend shape,
  rigged model и т.д.) при первом использовании, коротко
- Если предлагается решение, требующее регистрации/оплаты/новых
  аккаунтов — явно предупреждай заранее, не как сюрприз в середине шагов
- Изменения в архитектуре моста (протоколе) — всегда обсуждать перед
  реализацией, не молча менять формат
- Если шаг требует действий вне кода (GUI Godot, установка ПО) —
  пошаговая инструкция с указанием, куда нажимать
- Не предлагать Nvidia/CUDA-специфичные инструменты ни при каких
  обстоятельствах — железо физически не поддерживает их
